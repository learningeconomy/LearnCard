# AWS KMS Disaster Recovery Runbook

## Overview
This runbook outlines the procedures for responding to AWS KMS unavailability or key compromise. AWS KMS is a critical dependency for credential signing in the LearnCard network.

## Scenario 1: AWS KMS Service Unavailability
If the AWS KMS service becomes unavailable in our primary region, signing operations will fail.

### Immediate Actions
1. **Verify Status**: Check the [AWS Service Health Dashboard](https://health.aws.amazon.com/health/status) to confirm if there is a regional outage.
2. **Alert Stakeholders**: Notify the engineering and product teams about the service disruption.
3. **Contact AWS Support**: Open a high-priority support case with AWS if the outage is not already acknowledged.

### Recovery Procedure
- **Failover to Secondary Region**: If we have configured multi-region keys, update the application configuration to use the KMS endpoint in the secondary region.
- **Use Backup Key Material**: If we maintain offline backups of key material (e.g., for imported keys), follow the procedure to import this material into a new KMS key or a temporary signing service. Note that keys generated by KMS cannot be exported.

## Scenario 2: Key Compromise
If a KMS key is suspected of being compromised.

### Recovery Procedure
1. **Disable the Key**: Immediately disable the compromised key in the AWS KMS console to prevent further use.
2. **Rotate the Key**: Create a new KMS key and update the application configuration to use the new key for all future signing operations.
3. **Audit Usage**: Review AWS CloudTrail logs to identify any unauthorized signing operations performed with the compromised key.
4. **Revoke Credentials**: If unauthorized credentials were issued, use the LearnCard revocation system to invalidate them.

## Key Rotation Procedure
We follow the AWS best practice of annual key rotation.
1. **Enable Automatic Rotation**: For KMS-generated keys, enable automatic annual rotation in the AWS console.
2. **Manual Rotation**: For imported key material, manually generate a new key and update the application configuration.

## Testing Procedure
To ensure readiness, we perform disaster recovery drills in the staging environment quarterly.
1. **Simulate Outage**: Temporarily revoke the IAM permissions for the brain-service to access KMS.
2. **Verify Failure**: Confirm that signing operations fail with the expected error codes.
3. **Restore Access**: Restore the IAM permissions and verify that the service recovers correctly.
4. **Document Results**: Record the time to detect and the time to recover in the quarterly engineering report.
